# 总体架构与代码分析

## 文档维护规则

自 `2026-04-13` 起，本文件与 `开发规范与注释约定.md`、`修改记录.md` 必须同步维护。凡是模块职责、主入口、接口边界、状态机流程发生变化，本文件都要同步更新。

## 当前统一架构

### 顶层执行链

`PRG_MainControl`

1. 先在顶层生成固定 10ms 工艺节拍与固定 100ms 温控节拍，供后续主控块与温控块直接消费。
2. 再调用 `gvl_fbHMIParaInterface`，处理在线参数与 HMI 编辑缓冲区的同步。
3. 再调用顶层面板调度实例 `gvl_fbPanelDispatcher`，把原始面板 IO 解码为按键边界，并把关键按键先回灌到独立面板命令源 `gvl_stPanelCommandSource`。
4. 再调用顶层命令仲裁实例 `gvl_fbCommandArbiter`，先保留外部入口已经写入的基础公共命令，再叠加面板命令源负责的模式与手动字段，统一合成为最终 `gvl_stCommand`。
5. 最后调用顶层温控实例 `gvl_fbEK312` 和机器主控实例 `gvl_fbMachineController`，统一完成温控刷新、模式判定、自动流程调度、子模块执行和状态汇总。
6. `PRG_MainControl` 绑定 1ms 基础任务；主控工艺逻辑由顶层 10ms 节拍驱动；温控块每 1ms 都会被顶层调用，但只有命中顶层 100ms 节拍时才执行一轮温控计算。
7. 顶层入口和机器主控的当前维护基线已提升为“逐变量详细注释 + 逐参数注释 + 逐执行行可直接读懂”的注释密度。

### 主控内部分层

`FB_MachineController`

1. 读取 `ST_MachineCommand`、`ST_MachinePara`、`ST_MachineSensor`。
2. 内部调用：
   `FB_MachineModeResolver`、`FB_ManualMode`、`FB_MoldAdjustMode`、`FB_AutoCycle`、`FB_MachineCommandComposer`、`FB_Clamp`、`FB_MoldAdjust`、
   `FB_Nozzle`、`FB_Inject`、`FB_Meter`、`FB_Eject`、`FB_EjectMode`、`FB_Core`、`FB_MachineStatusAggregator`。
3. 工艺路由、模式切换和自动流程只消费顶层送入的 10ms 节拍；执行模块与其内部轴层仍随 1ms 基础任务运行。
4. 8 个执行模块各自在模块内部持有 `FB_HydAxis`，主控只做工艺层命令分发和模块状态汇总。
5. 温控系统由顶层独立运行并只消费顶层送入的 100ms 节拍；`FB_MachineController` 不再保留 `stSensor.bTempReady` 这类温控前提接口，也不处理温控块接口和温控状态。
6. `PRG_MainControl.st` 与 `FB_MachineController.st` 现作为注释密度的新基线，要求变量定义和关键执行行保持详细注释。

### 全局边界

1. `GlobalVars.st` 只保留在线运行必需的共享结构。
2. `Instances.st` 只保留 `gvl_fbHMIParaInterface`、`gvl_fbPanelDispatcher`、`gvl_fbCommandArbiter`、`gvl_fbEK312`、`gvl_fbMachineController` 五个顶层实例。
3. 旧版手动/自动中间缓存已经移除，避免旧架构与新架构同时生效。
4. 当前工程中的全局共享对象前缀已统一收口为 `gvl_`。
5. `gvl_ePanelModel`、`gvl_stPanelIO`、`gvl_stPanelButtons`、`gvl_stPanelCommandSource` 共同组成顶层面板边界，分别负责型号选择、原始 IO 交互、综合按键/按灯状态和面板命令源锁存。

## 模块职责分析

### HMI 系统

文件：`Code/HMI系统/FB_HMIParaInterface.st`

职责：

1. 管理在线参数与 HMI 编辑缓冲区之间的读回、下发和脏标记。
2. 在机器忙碌时阻止 HMI 直接下发在线工艺参数。
3. 为 HMI 提供统一状态反馈。

### 主程序入口

文件：`Code/主程序入口/*.st`

职责：

1. `PRG_MainControl` 是唯一有效主入口。
2. `MachineControlTypes.st` 定义机器级统一接口。
3. `FB_CommandArbiter.st` 负责先保留基础公共命令，再把面板命令源负责的模式与手动字段叠加为最终机器命令。
4. `FB_MachineController.st` 是整机唯一主控编排器，但内部已按“模式解析、命令合成、状态汇总”拆成辅助层。
5. `PRG_MainControl.st` 直接调用顶层 `gvl_fbPanelDispatcher`、`gvl_fbCommandArbiter`、`gvl_fbEK312` 并生成 10ms/100ms 节拍；`FB_MachineController.st` 只消费顶层 10ms 节拍，不再驱动、消费或汇总温控块本体状态。

### 面板管理

文件：`Code/面板管理/*`

职责：

1. `PanelTypes.st` 定义 `ST_PanelButton`、`ST_PanelButtons`、`ST_PanelIO` 与 `E_PanelModel`，统一收口逻辑排序号、真实输入位、原始 IO 和面板型号。
2. `FB_MK110`、`FB_MK116`、`FB_MK150` 现在统一为同一写法：各型号块只维护当前型号的 `uiPanelOrder` 与真实输入字/位定义，不再直接读键或写灯。
3. 公共类型与公共辅助函数统一放在 `Code/面板管理/面板公用/*`，供各面板型号复用。
4. `F_PanelReadKeyByOrder` 现已改为按型号块写入的真实输入字/位读取单个逻辑键，`FB_PanelReader` 负责批量回写 `stPanelButtons.*.bValue`，`FB_PanelLampEncoder` 负责按 `uiPanelOrder` 汇总 `uiOUT0..uiOUT3`。
5. `FB_Panel` 现作为兼容封装块存在，内部串联 `FB_PanelReader` 与 `FB_PanelLampEncoder`；新架构优先在调度层分开调用“读键”和“写灯”。
6. 面板灯输出统一通过 `FB_PanelLampWriter` 按 `uiPanelOrder` 写入 `uiOUT0..uiOUT3`，保证同一逻辑键的键值和灯值始终绑定到同一个逻辑排序号。
7. `FB_PanelDispatcher` 统一负责硬件面板型号分发、原始 IO 调度、关键按键回灌面板命令源和灯值回写。
8. 面板输入字、输出字与位操作辅助接口统一采用 `UINT` 类型，并统一使用 `uiINx`、`uiOUTx`、`uiValue` 前缀，与整型命名规范保持一致。
9. 当前三个型号块都已收口为“逻辑排序号只在型号块里定义一次，输入物理映射也只在型号块里定义一次”；后续切换面板型号或调整键位顺序时，只需要修改对应 `FB_MK110`、`FB_MK116`、`FB_MK150` 中的定义。
10. `FB_PanelDispatcher` 当前采用“本拍先写型号映射并读键，再更新命令源与灯请求，最后由独立灯编码层生成原始输出字”的顺序，避免面板灯输出沿用上一拍残留值，也不再双调用同一型号块。

### 模式管理

文件：`Code/模式管理/*`

职责：

1. `FB_ManualMode`：把机器手动命令路由到各执行模块。
2. `FB_MoldAdjustMode`：把调模模式下允许的动作收口，并强制锁模进入调试方向。
3. `FB_AutoCycle`：只做自动流程步骤调度，不直接驱动轴；自动前置条件当前只检查门禁、液压和系统无报警。
4. `PRG_*` 文件现为兼容占位，不再承担实际执行。

### 伺服系统

文件：`Code/伺服系统/液压轴/*`

职责：

1. `FB_HydAxis`：对模块写入的 `ST_AxisRefHyd` 做最终轴实例化。
2. 统一完成轴号绑定、能力上限回写、命令限幅和轴层报警叠加。
3. 当前 8 个执行模块都在模块内部各自实例化一份 `FB_HydAxis`：
   合模、调模、座台、射胶、储料、托模基础、托模模式、中子。
4. `AxisTypes.st` 新增了 `ST_HydAxisConfig`，用于各子模块内部统一收口轴号和保护边界。
5. `CommonTypes.st` 中的 `ST_AxisRefHyd` 已收口为液压执行链当前真实使用的最小公共边界，只保留轴参与状态、能力边界、目标命令、实际反馈、泵请求和故障码，不再混放旧版伺服全量字段。

### 合模系统

文件：`Code/合模系统/*`

职责：

1. `FB_Clamp`：处理开模、合模、低压、高压、调试等动作。
2. `FB_MoldAdjust`：处理模厚调节进退动作。
3. 对应 `*Types.st` 文件维护参数结构与状态枚举。

### 注射系统

文件：`Code/注射系统/*`

职责：

1. `FB_Nozzle`：座台进退控制。
2. `FB_Inject`：射出、保压、射退控制。
3. `FB_Meter`：储料动作控制。
4. 以上 3 个执行块已在 `2026-04-14` 补齐局部变量语义和状态机段落注释，后续修改时应保持同等注释粒度。

### 顶出系统

文件：`Code/顶出系统/*`

职责：

1. `FB_Eject`：基础托模进、保压、退动作。
2. `FB_EjectMode`：定次、震动、停留等组合托模模式。
3. `FB_Core`：中子进退动作。
4. `FB_EjectMode` 通过内部 `FB_Eject` 实例直接组合更高层托模动作，不再额外重复执行一层托模轴层。
5. 以上 3 个执行块已在 `2026-04-14` 补齐局部变量语义和状态机段落注释，后续修改时应保持同等注释粒度。

### 温度系统

文件：`Code/温度系统/EK312/*`

职责：

1. `FB_EK312`：12 路温控的仿真/控制接口。
2. 由 `PRG_MainControl` 顶层每 1ms 直接调用，并消费顶层下发的 100ms 刷新拍执行一轮温控计算。
3. 温控块实例放在顶层实例区，由主入口直接驱动，不再下沉到机器主控内部，也不再把温控块状态并入机器主控。

## 本次统一后的关键结论

### 完整性

1. 类型定义、功能块定义和引用关系已做静态核对，当前 `Code` 目录内没有发现缺失的自定义类型或功能块引用。
2. 顶层架构已统一为“单入口 + 单主控”。
3. 新增了独立轴功能块层，执行模块与最终轴实例化已经解耦。
4. 旧版多入口残留逻辑已收敛为兼容占位，避免重复驱动。
5. 全局变量、全局实例命名已统一为 `gvl_` 前缀，顶层边界命名更明确。

### 可执行性

1. 已修复三处明确的 ST 语法问题：`FB_Clamp.st`、`FB_MachineController.st`、`FB_EK312.st` 中的 `END_FOR` 已补齐分号。
2. `PRG_MainControl` 现在已包含 HMI 同步和机器主控调用，单独绑定即可形成完整执行链。
3. `FB_Clamp`、`FB_MoldAdjust`、`FB_Nozzle`、`FB_Inject`、`FB_Meter`、`FB_Eject`、`FB_EjectMode`、`FB_Core` 使用到的轴，已经下沉到各执行模块内部实例化，主控不再处理轴配置、轴使能和轴状态并入。
4. `FB_MachineController` 已改为消费 `PRG_MainControl` 顶层生成的 10ms 逻辑节拍；在 1ms 基础任务下，轴层和执行块可保持高频刷新，自动流程和模式路由按 10ms 刷新。
5. `FB_EK312` 已提升到 `PRG_MainControl` 顶层直接调用，并由顶层 100ms 节拍驱动刷新；`FB_MachineController` 不再消费或汇总温控块结果。
6. `FB_AutoCycle.st` 已补回本地 `VAR` 状态缓存声明，自动流程块恢复完整的功能块声明结构。
7. `FB_EK312.st` 在停机态、空闲态和故障态都会主动清零全部温控输出地址镜像，且只消费顶层下发的 100ms 刷新节拍。
8. 主干 `.st` 文件已统一整理为 `UTF-8 with BOM`，文件头、接口注释和类型注释的显示一致性已收口。
9. 由于当前工作区不包含 PLC 工程任务配置文件，也没有实际 ST 编译器，本项目尚未做编译级验证；1ms 基础任务绑定仍需在工程环境中确认。
10. `FB_Clamp`、`FB_MoldAdjust`、`FB_Nozzle`、`FB_Inject`、`FB_Meter`、`FB_Eject`、`FB_EjectMode`、`FB_Core` 已补齐局部变量定义说明、公共分支意图和状态机分段注释，便于后续按状态排查动作异常。
11. `PRG_MainControl` 与 `FB_MachineController` 已开始按“全部 `VAR_*` 变量必须详细注释、顶层调用参数逐行注释”的新规则收口，后续主干文件按同一标准继续补齐。
12. 温控块接口已进一步简化，`FB_EK312` 删除 `bManualMode` 与 `uiActTime`，主控侧同步删除温控镜像输入、局部缓存和机器级温控状态汇总。
13. 面板管理层已新增 `ST_PanelButtons`、`ST_PanelIO`、`E_PanelModel`、`FB_MK110`、`FB_MK116`、`FB_MK150`、`FB_PanelReader`、`FB_PanelLampEncoder` 与 `FB_PanelDispatcher`；三个型号块现已统一为“只维护 `uiPanelOrder` 与真实输入字/位”的结构，后续切换面板型号时只需要调整对应型号块中的定义。
14. `Code/伺服系统/CommonTypes.st` 已完成液压轴公共边界瘦身，移除当前工程未消费的旧伺服大结构字段，后续液压模块与轴层都以精简后的 `ST_AxisRefHyd` 作为统一交互边界。
15. `PRG_MainControl` 已补上顶层面板调度、命令仲裁与温控真实调用链：`gvl_fbPanelDispatcher` 每拍解码面板并回灌面板命令源，`gvl_fbCommandArbiter` 先保留基础公共命令再叠加面板负责的模式与手动字段，`gvl_fbEK312` 每拍都被调用但只在 100ms 节拍上执行温控计算。
16. 面板读键层已从“排序号线性映射输入位”改为“型号块显式提供输入字号/位号”的实现，当前三种面板的真实输入分布都能在公共层正确表达。
17. `FB_MachineController` 已完成首轮内部拆层：模式解析下沉到 `FB_MachineModeResolver`，命令合成下沉到 `FB_MachineCommandComposer`，状态汇总下沉到 `FB_MachineStatusAggregator`，主控本体主要保留 10ms 路由调度与模块执行。

### 架构风险

1. 如果 PLC 工程里仍把旧版 `PRG_MachineController`、`PRG_ManualMode`、`PRG_MoldAdjustMode`、`PRG_AutoCycle` 绑定到任务，会与新架构说明冲突。
2. 本次已把这些程序改为兼容占位，但工程层仍应检查任务映射，确保只运行 1ms 基础任务上的 `PRG_MainControl`。
3. `Code/IO管理` 当前仍缺少真实现场 IO 绑定实现，面板原始 IO、机器传感器与最终输出边界虽然已经收口，但现场“采集/下发”链路还没有在仓库内闭环。
4. 当前仓库内仍未看到独立的 `bStart`、`bStop`、`bReset`、`bEStop` 实际命令来源接线；面板层目前主要闭环了模式切换、温控开关和点动命令，后续仍需补充启动/停止源的实际接入边界。

## 后续维护建议

1. 新增模块时，先扩展 `MachineControlTypes.st` 和 `GlobalVars.st` 的统一边界，再扩展 `FB_MachineController` 与对应辅助层。
2. 修改任何状态机时，必须同步更新本文件中的职责说明和执行链描述。
3. 若后续继续扩展主控功能，优先扩展 `FB_MachineModeResolver`、`FB_MachineCommandComposer`、`FB_MachineStatusAggregator` 三层，不要把辅助层逻辑重新回填到 `FB_MachineController` 本体。
4. 若后续接入真实 PLC 工程文件，建议补一份“任务配置与下载说明”文档，并在本文件追加编译验证记录。
