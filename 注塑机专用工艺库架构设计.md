# 注塑机专用工艺库（IM_Lib）架构设计与功能块封装指导

## 1. 背景与核心问题：为何需要注塑机专用库？

在通用运动控制（PLCopen 标准）中，基础功能块 `MC_MoveAbsolute` 的语义是：“以指定的速度（Velocity）移动到绝对位置（Position）”。

然而，基于**注塑机业务模型**，真正的“合模（Mold Close）”动作并非简单的点位移动，而是一个包含多段工艺参数和严格时序的**复合动作**：
1. **快合模段（Fast Close）**：大流量快速移动，节约机器周期时间。
2. **低压模保段（Mold Protection）**：低速、极低压力。一旦遇到异物（如未脱落的塑料件或顶针未退回），油缸无法克服阻力，立刻停止并报警，保护昂贵的模具。
3. **高压锁模段（High Pressure Lock）**：模具闭合后，用极高的系统压力建立锁模力（Tonnage），防止射胶时涨模。

**结论**：如果让工艺层（用户代码）直接用多个 `MC_MoveAbsolute` 和 `MC_TorqueControl` 来拼凑合模动作，会带来极大风险：
- **实时性差**：工艺层运行在较高周期的任务中，当模保碰伤触发时，从发现异常到通知底层停止可能延迟数毫秒至数十毫秒，模具已经被压坏。
- **代码冗余**：每次换机型或修改工艺流程，都需要重新编写状态机。
- **违背解耦**：工艺工程师被迫去处理底层的液压阀切换和压力转换逻辑。

因此，**必须建立一个基于 `HydroMotionLib` 之上的“注塑机专用工艺库（Injection Machine Library, 简称 IM_Lib）”**。

---

## 2. 分层解耦的软件工程设计

结合 L1-L5 分层架构，注塑机专机逻辑与通用液压控制必须做清晰的隔离。整体系统应分为两个平行的代码库（或命名空间）：

### 2.1 基础通用层：HydroMotionLib (L1-L5)
负责纯粹的液压运动与基础 PLCopen 接口。
- 只有点位、速度、压力控制。
- 不懂什么是“合模”、什么是“射胶”，只知道 `Axis` 和 `Strategy`。

### 2.2 注塑专用层：InjectionMachineLib (建立在 L5 之上)
对注塑机的具体工艺节点进行高级封装，提供针对性的功能块（FB）。
- **工艺功能块**：`IM_MoldClose`（合模）, `IM_MoldOpen`（开模）, `IM_Inject`（射胶）, `IM_Eject`（顶出）。
- 这些专机 FB 在内部持有各段的设定参数，并将其转化为底层 `HydroMotionLib` 可以理解的策略配置和基础指令。

## 3. 核心功能块 `IM_MoldClose` 设计建议

### 3.1 工艺参数数据结构 (Profile)
工艺层不再下发单一点位，而是下发一条完整的**合模工艺曲线**（Recipe/Profile）。

```c
// 合模工艺曲线参数结构体
typedef struct {
    // 1. 快合段
    float fast_close_pos;      // 快合结束位置
    float fast_close_vel;      // 快合速度
    
    // 2. 模保段
    float mold_protect_pos;    // 模保结束（接触）位置
    float mold_protect_vel;    // 模保慢速
    float mold_protect_press;  // 模保压力极限 (极低)
    
    // 3. 高压锁模段
    float high_pressure_bar;   // 高压锁模目标压力
    float lock_time_s;         // 锁模建压保持时间
} IM_MoldCloseProfile;
```

### 3.2 功能块接口定义
采用类似 IEC 61131-3 功能块的输入输出引脚风格。

```c
typedef struct {
    // ---- 输入引脚 (Inputs) ----
    bool Execute;                    // 触发沿执行
    HydroAxis* Axis;                 // 绑定的基础液压轴
    IM_MoldCloseProfile Profile;     // 合模工艺参数
    
    // ---- 输出引脚 (Outputs) ----
    bool Done;                       // 动作完成且高压建立
    bool Busy;                       // 动作执行中
    bool Active;                     // FB当前控制着该轴
    bool CommandAborted;             // 被其他命令打断
    bool Error;                      // 发生错误（如模保失败）
    int  ErrorID;                    // 错误码
    
    // ---- 专机状态指示 ----
    int  CurrentPhase;               // 当前执行的阶段（快合=1, 模保=2, 建压=3）
    
    // ---- 内部私有状态 (Opaque) ----
    void* internal_state;            
} IM_MoldClose_FB;

// FB 执行主函数（放在工艺层的主循环中调用）
void IM_MoldClose(IM_MoldClose_FB* fb);
```

### 3.3 内部逻辑流转与底层交互
当工艺层调用 `IM_MoldClose` 时，其内部运作逻辑如下：
1. **参数透传**：FB 将 `Profile` 中的分段数据，通过底层的私有接口配置给 `HydroAxis` 的 `ClampStrategy`（合模策略）。
2. **启动命令**：FB 内部调用底层基础 API `MC_MoveAbsolute(Axis, ...)`，甚至调用扩展的 `MC_MoveProfile` 让底层知道这是一个多段动作。
3. **状态接管**：FB 开始监听 `Axis` 的 `CurrentPhase` 或当前位置信息。
    - **快进 -> 模保 的切换**：由 L3 层 `ClampStrategy` 硬件中断或微秒级循环在底层**自动**完成，FB 只负责将切换状态反映给 `CurrentPhase` 输出引脚。
    - **模保故障拦截**：如果发生模保报警（阻力超限），L3 层策略立即切断输出，轴进入 `ErrorStop` 状态；FB 捕获到该状态后，输出引脚 `Error = true`, `ErrorID = ERR_MOLD_PROTECT_FAIL`。

## 4. 总结：开发建议路线图

基于上述分析，接下来的开发路线建议如下：

1. **坚持底层不带工艺黑话**：`HydroMotionLib` 里面不出现 `Mold`, `Inject` 等注塑专用词汇，而是统称为 `Axis`, `Strategy`, `Phase`。
2. **建立专机库模块**：在项目中新建 `InjectionMachineLib` 目录，封装诸如 `IM_MoldClose`, `IM_Inject` 等高级 FB。
3. **工艺层极简调用**：注塑机的工艺逻辑代码（通常是跑在状态机或 PLC 梯形图上的代码）**只能实例化并调用 IM_ 专机功能块**，绝对禁止在工艺代码里直接操作阀口开关或用基础 `MC_` 函数拼凑合模。
4. **底层保障安全性**：像“模保”这种人命关天、设备关天的微秒级保护，必须放在 `HydroMotionLib` 的 `L3 Strategy` 层里实现硬核阻断，`IM_MoldClose` FB 只负责参数搬运和状态监控。

这样做，工艺工程师只需关心工艺参数表（Profile），系统架构师关心 FB 流转，控制算法工程师关心 L3 的油压建压与阀控时序，完美实现**现代工业软件工程的边界解耦**。
